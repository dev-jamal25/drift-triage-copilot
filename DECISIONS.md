# Design Decisions

Each decision: the choice, alternatives considered, and the reason.

## Webhook vs polling for drift events
Webhook from platform to agent on every severity change. Agent persists each
incoming event to a `drift_events` table before processing, and on startup
re-reads any unprocessed events from that table. This gives webhook latency
in the happy path and polling-style durability if the agent is down when an
event fires.

## LLM model choice (per sub-agent)
- Triage: claude-haiku-4-5-20251001 (cheap, structured-output classification)
- Comms: claude-haiku-4-5-20251001 (cheap, summarization only)
- Action: claude-sonnet-4-5 (stronger reasoning needed for multi-step
  decisions about what to do with the model)
Rationale: tier by complexity. Triage and comms are pattern-match tasks where
Haiku is more than enough; the action sub-agent reasons about consequences,
which is where stronger model earns its cost.

## Drift severity thresholds (PSI, chi²)
- PSI: <0.1 green, 0.1–0.25 yellow, >0.25 red (industry-standard bins)
- Chi²: p<0.01 yellow, p<0.001 red
- Output drift (predicted positive class): same PSI bins as above
- Overall severity = max severity across all features and output drift

## Rolling window: count-based vs time-based
Last 1000 predictions, recomputed every 60 seconds. Count-based gives
stable sample sizes for PSI/chi² statistics regardless of traffic
fluctuations; a time-based window would shrink to noise during low traffic.

## Idempotency strategy in the worker
Idempotency key format: `{investigation_id}:{action_type}:{target_version}`,
generated by the agent at enqueue time and embedded in every QueuedAction.
Worker checks `actions_log` for an existing row with the same key before
executing; if status is `success`, returns cached result; if `failure` with
attempts exhausted, returns DLQ. We run a single worker process to sidestep
cross-worker race conditions — documented as a known scaling limitation
acceptable for the sprint.

## How agent recovers when checkpoint references a stale model URI
On checkpoint resume, the supervisor validates the model URI against the live registry before proceeding. Stale investigations are closed with a logged explanation rather than acted upon.

## How "superseded" approvals are detected and rejected
Superseded approvals are detected at enqueue time by checking for open approvals against the same model URI, and again at approval time before the agent acts. An approval is rejected if its superseded_by field is non-null or if a newer drift event for the same model arrived after the approval was created.

## Manual promotion bypass (when allowed, how audited)
Yes, allow it. The /promote endpoint accepts two auth modes: an agent-issued approval_token (normal flow) OR an X-Admin-Bypass-Token header (emergency flow). Both paths log to a single promotions_audit table with a source column (agent | manual_bypass) plus the requestor identity. Dashboard surfaces manual bypasses with a yellow banner.
