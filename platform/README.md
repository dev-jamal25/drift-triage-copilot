## dataset, prep rules, model, and threshold
Use `bank-additional-full.csv` (the full UCI Bank Marketing additional release, ~41k rows × 21 cols, `;`-separated) over the smaller `bank-additional.csv` because PSI/chi² drift detection in later slices needs a reference distribution with enough categorical mass to be statistically meaningful. Data-prep rules locked by tests in `platform/tests/test_data.py`: drop `duration` (it leaks the call outcome — known target leak per the UCI data card); map `y` → `target` int {0, 1}; keep `"unknown"` as a literal categorical level rather than imputing it, since "unknown" is itself signal in this domain (e.g. self-reported loan status); encode `pdays == 999` (sentinel for "never previously contacted") as two derived columns `was_previously_contacted` and `pdays_clean`, dropping the original `pdays`; stratified 60/20/20 split on `target` with `random_state=42`. Bake-off across `LogisticRegression`, `RandomForestClassifier`, `GradientBoostingClassifier` (5-fold stratified CV, `class_weight="balanced"` where supported — GB doesn't, so we lean on threshold tuning for it). Selected by mean `f1_macro` because the dataset is ~11% positive and macro F1 weights both classes equally; on this run logreg won (0.6697 vs. 0.6482 GB and 0.6416 RF), and all three trounce the majority-class baseline (0.4702). Threshold tuned on the validation set as the **highest** value where recall ≥ 0.75 — chosen this rule because in marketing-call recall, missing buyers is the dominant cost; max-of-qualifying picks the most precise threshold subject to the recall floor (val 0.38 → val recall 0.754 / precision 0.240; held-out test mirrors at recall 0.754 / precision 0.241 / ROC-AUC 0.801). MLflow registration is intentionally deferred to Slice 2.

## MLflow artifact triple, registry alias, and load-back fidelity check
Each training run logs three artifacts in one MLflow run: the fitted sklearn pipeline (via `mlflow.sklearn.log_model` with an inferred signature and a 5-row input example), a custom `schema.json` describing input columns + categorical levels (pulled from the fitted `OneHotEncoder`) + output shape + class labels, and a `model_card.md` rendered from `platform/ml/model_card.md` with concrete metrics and dataset hash filled in. The model is auto-registered via `registered_model_name=` and assigned the `staging` alias on the new version (instead of the legacy `transition_model_version_stage` API, which MLflow 3 removed). The platform's external `PromotionStage` Literal — `"Staging" | "Production" | "Archived"` — is preserved by mapping each name to a lowercase alias inside `ml/registry.py`; nothing changes in `shared/contracts.py`. Two run-level tags carry provenance: `dataset_hash` (sha256 of the cleaned dataframe's parquet bytes — deterministic for the same source CSV) and `env.{python,sklearn,mlflow,pandas,numpy,platform}`, so a stale agent checkpoint can be cross-checked against current registry state when the supervisor resumes (per the "stale model URI" decision above). After registering, the CLI loads the model back via `models:/<name>@staging` and runs a fidelity check (`max|Δp| ≤ 1e-9` over 200 val rows); a Slice-2 run on the full UCI dataset reports `max_abs_diff=0.0` — the round-trip is float-identical, which closes the obvious source of drift between training and serving. Default tracking URI is `file:./mlruns` for local dev; the production docker-compose `MLFLOW_TRACKING_URI=http://mlflow:5000` is wired in via the `--mlflow-uri` flag and will move into `pydantic-settings` in Slice 3 alongside the FastAPI lifespan loader.

## Model loading, threshold sourcing, and prediction schema location (platform service)
The fitted classifier is loaded once during the FastAPI lifespan (wrapped in `asyncio.to_thread` so the event loop isn't blocked on startup) and stashed on `app.state.model_bundle`; the request path reaches it via a `Depends(get_model_bundle)` provider that returns 503 if the bundle isn't there yet (instead of 500), so Kubernetes-style readiness probes can drive traffic correctly. Threshold is *not* a separate config knob — `model_loader.load_bundle` reads it from the originating MLflow run's params (where `ml/registry.py` logged it during registration), so the registry remains the single source of truth and there's no second file to drift out of sync. Settings flow through one `pydantic_settings.BaseSettings` in `app/core/config.py` (`PLATFORM_*` env-var prefix, `protected_namespaces=()` to allow the `model_*` field names) — `os.getenv` is forbidden anywhere else per CLAUDE.md. The prediction request schema lives in `platform/app/schemas/prediction.py`, **not** in `shared/contracts.py`: the predict API is the platform's public boundary, not a cross-service wire contract, and locking it inside `shared/` would force every PR that adjusts a feature column to be co-signed by the agent track. Field aliases match raw UCI column names with dots (`emp.var.rate`, `cons.price.idx`, `nr.employed`) so a request body mirrors a row of the training CSV; `populate_by_name=True` lets clients send snake_case alternatives. `extra="forbid"` ensures `duration` is rejected at the API edge — the leakage column never reaches the model. `apply_pdays_sentinel` was extracted from `ml/data.clean` so the same `pdays==999 → (was_previously_contacted, pdays_clean)` transformation runs in the route as in training; this is the obvious source of training/serving skew, and routing both code paths through one function eliminates it. `predict_proba` is dispatched via `await asyncio.to_thread(...)` to honour the "no blocking I/O in request paths" rule even though sklearn is fast — the convention matters more than the millisecond. `/healthz` and `/readyz` distinguish liveness from model-readiness so the orchestrator can route traffic only when the model is actually available.
